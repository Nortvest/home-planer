# Домашний планёр

Две независимые части:

- **[backend/](backend/README.md)** — FastAPI-сервер, чистая архитектура, SQLite.
- **[frontend/](frontend/README.md)** — HTML/CSS/JS, без бандлеров и фреймворков.

Прочитайте соответствующий README для запуска.

## Быстрый старт

```bash
# Бэкенд
cd backend
uv sync --extra dev
uvicorn src.main:app --host 0.0.0.0 --port 8000 --reload

# Фронтенд
cd frontend
python -m http.server 5500
```

Затем откройте `http://localhost:5500` в браузере.

## Запуск

### Отдельный запуск (рекомендуется для разработки)

1. Поднимите бэкенд (порт 8000):
   ```bash
   cd backend && uvicorn src.main:app --host 0.0.0.0 --port 8000 --reload
   ```
2. Поднимите фронтенд (порт 5500):
   ```bash
   cd frontend && python -m http.server 5500
   ```
3. Убедитесь, что `frontend/js/config.js` указывает на `http://localhost:8000/api`.
4. Откройте `http://localhost:5500` в браузере.

### Объединённый запуск (для продакшена)

Бэкенд отдаёт статику фронта через один порт:

```bash
cd backend
SERVE_FRONTEND=1 uvicorn src.main:app --host 0.0.0.0 --port 8000
```

Откройте `http://localhost:8000`. Бэкенд автоматически монтирует каталог `frontend/` как статику.

### Первоначальная настройка

При первом запуске база данных создаётся автоматически в `data/planner.db`. Для быстрого заполнения демо-данными:

```bash
cd backend
uv run python -m src.infrastructure.db.seed
```

Скрипт создаст двух пользователей (Анна, Борис) и несколько базовых шаблонов задач.

## Конфигурация

### Бэкенд

Создайте `backend/.env` рядом с `pyproject.toml`:

```ini
DB_PATH=../data/planner.db
HOST=0.0.0.0
PORT=8000
TZ=Europe/Moscow
SERVE_FRONTEND=0
CORS_ALLOW_ORIGINS=*
FRONTEND_DIR=../frontend
```

| Переменная | Описание | По умолчанию |
|---|---|---|
| `DB_PATH` | Путь к SQLite-файлу | `../data/planner.db` |
| `HOST` | Хост для uvicorn | `0.0.0.0` |
| `PORT` | Порт для uvicorn | `8000` |
| `TZ` | Часовой пояс | системный |
| `SERVE_FRONTEND` | Отдавать статику фронта (`1`/`0`) | `0` |
| `CORS_ALLOW_ORIGINS` | Разрешённые CORS-источники | `*` |
| `FRONTEND_DIR` | Путь к каталогу фронта | `../frontend` |

### Фронтенд

Единственная настройка — адрес бэка в `frontend/js/config.js`:

```js
const API_BASE = 'http://localhost:8000/api';
```

В объединённом режиме (SERVE_FRONTEND=1) достаточно:

```js
const API_BASE = '/api';
```

## Docker

Самый простой способ запустить проект — через Docker Compose. Оба сервиса поднимаются одной командой, база данных сохраняется в `./data/`.

```bash
docker compose up -d
```

После запуска:
- Фронтенд: `http://localhost:5500`
- Бэкенд API: `http://localhost:8000/api`

`config.js` во фронте должен указывать на адрес бэка. В compose-сети это `http://backend:8000/api`, при доступе с хоста — `http://localhost:8000/api`.

### Dev-режим

`docker-compose.override.yml` подхватывается автоматически и монтирует исходники бэка с hot-reload:

```bash
docker compose up -d
```

### Остановка и данные

```bash
docker compose down
```

Данные сохраняются в `./data/` и переживают пересоздание контейнеров.

### Конфигурация

Скопируйте `.env.example` в `.env` и настройте переменные:

```bash
cp .env.example .env
```

| Переменная | Описание | По умолчанию |
|---|---|---|
| `DB_PATH` | Путь к БД внутри бэка | `/app/data/planner.db` |
| `TZ` | Часовой пояс | `UTC` |
| `BACKEND_PORT` | Порт бэка на хосте | `8000` |
| `FRONTEND_PORT` | Порт фронта на хосте | `5500` |

## Известные ограничения

- **Отсутствует аутентификация**: приложение предназначено для домашней сети (LAN). Любой, кто имеет доступ к странице, может выполнять любые действия.
- **Нет реалтайм-синхронизации**: изменения, внесённые одним пользователем, видны другим только после обновления страницы.
- **SQLite как хранилище**: база данных — единый файл, что подходит для домашнего использования, но ограничивает конкуррентную запись при высокой нагрузке.
- **Бэкап вручную**: автоматический бэкап базы данных не реализован. Рекомендуется периодически копировать `data/planner.db`.
- **Нет поддержки мульти-дома**: один планер на одну семью. Несколько независимых планов не поддерживаются.
- **Статусы задач считаются динамически**: при каждом запросе статус вычисляется на fly. Это означает, что «просрочено» определяется моментом запроса, а не заранее.

## Идеи развития

- Экспорт баланса за период в CSV.
- Подписка на RSS/Atom с просроченными задачами.
- Push/email уведомления о просрочках.
- Заметки/комментарии на карточке задачи.
- Aчивки за активность («неделя без просрочек», «баланс 50/50»).
- WebSocket-уведомления для обновления без перезагрузки.
- PWA с оффлайн-кэшем календаря.
- Бэкап базы данных по кнопке из админки.
- Идентификация пользователей через сессию (для защиты от случайных действий).
- Иерархические задачи (подзадачи внутри задачи).
